# spec.md — 오늘 뭐먹지?

> **한 줄 요약:** 직장 팀이 점심 메뉴를 제안·투표로 빠르고 공평하게 정하고, 결과와 최근 이력을 한곳에서 본다.

개발 시 기술 스택·제약은 저장소 루트 **`AGENTS.md`** 를 따른다. (Next.js App Router, TypeScript, Tailwind, Supabase)

---

## 1. 배경·목표

- **문제:** 매일 점심 메뉴를 정할 때 눈치 보기, 결정 지연, 불공평해 보이는 선택.
- **목표:** 같은 팀원이 **제안 → 투표 → 자동 집계 → 오늘의 메뉴 공유 → 이력 확인**까지 한 흐름으로 끝낸다.
- **범위:** MVP는 **점심 1회차(하루 1세션)** 기준으로 단순하게. 복잡한 권한·결제·배달 연동은 나중.

---

## 2. 사용자

- **팀원:** 로그인(또는 팀 참여) 후 제안·투표·결과·이력 조회.
- **역할:** MVP에서는 **모두 동일 권한**으로 충분. (팀장 전용 “강제 마감” 등은 선택 사항)

---

## 3. 핵심 기능 (MVP)

| ID | 기능 | 설명 |
|----|------|------|
| F1 | 메뉴 제안 | 팀원이 오늘 후보로 먹고 싶은 메뉴를 자유롭게 추가한다. (중복 제안 규칙은 아래 정책 참고) |
| F2 | 투표 | 제안된 후보에 **1인 1표**로 투표한다. 마감 전에는 **투표 변경**을 허용할지 여부만 정하면 된다. (권장: **변경 허용**) |
| F3 | 자동 선정 | **마감 시점**에 최다 득표 메뉴를 **오늘의 결정 메뉴**로 확정한다. **동점**이면 정책을 하나 정한다. (권장: 동시 공동 1위 표시 또는 먼저 제안된 항목 우선 — 구현 시 한 가지로 통일) |
| F4 | 결과 공유 | 확정된 **오늘의 메뉴**를 팀 화면 상단(또는 전용 영역)에 **모든 팀원이 동일하게** 볼 수 있다. |
| F5 | 최근 이력 | **과거에 확정된 메뉴**를 날짜순으로 본다. (최근 N일 또는 스크롤 목록 수준으로 단순화) |

---

## 4. 동작 흐름 (하루 기준)

1. 팀이 **오늘의 점심 세션**을 연다. (또는 자정·첫 접속 시 자동 생성 — 구현 편한 쪽 선택)
2. 팀원이 후보 **제안** (F1).
3. 팀원이 후보에 **투표** (F2). 마감 전까지 참여.
4. **마감** (수동 버튼 또는 정해진 시각). 서버/DB 기준으로 집계해 **결정 메뉴 확정** (F3).
5. 화면에 **오늘의 메뉴** 표시 (F4).
6. **이력**에 오늘 결과가 쌓임 (F5).

---

## 5. 정책·엣지 케이스 (구현 시 명시)

- **1인 1표:** 한 사용자는 활성 세션에서 **투표는 하나의 후보에만** 연결된다.
- **중복 제안:** 같은 이름 메뉴를 여러 명이 제안할지 — **권장:** 문자열 정규화 후 합치거나, “같은 이름이면 득표 합산” 등 한 규칙만 택한다.
- **후보 0개·투표 0표:** 마감 시 “미정” 처리할지, 마감 불가로 할지 **한 가지**로 정한다.
- **타임존:** “오늘”의 기준은 **팀 또는 서버 기준 타임존 하나**로 고정한다. (MVP는 한국 시간 단일이어도 됨)

---

## 6. 데이터 모델 초안 (Supabase용 가이드)

AI/개발자가 테이블을 설계할 때 참고. 이름은 예시이며 실제 마이그레이션에서 조정 가능.

| 엔티티 | 용도 |
|--------|------|
| `teams` | 팀(또는 워크스페이스) |
| `profiles` / `users` | Supabase Auth 사용자와 매핑되는 표시명 등 |
| `team_members` | 사용자 ↔ 팀 소속 |
| `lunch_sessions` | 날짜(또는 날짜+팀) 단위 세션, 상태: `open` / `closed`, 마감 시각 |
| `menu_suggestions` | 세션별 제안 메뉴 텍스트, 제안자 |
| `votes` | 세션당 사용자 1행(또는 유니크 제약으로 1표 보장), 선택한 `menu_suggestion` |
| `session_results` | 마감 시 확정 메뉴(들) 스냅샷 — 이력(F5) 조회용 |

---

## 7. 화면(페이지) 후보 — MVP

- **팀 홈 / 오늘 점심:** 제안 목록, 투표 UI, 마감 상태, 오늘의 결정 메뉴.
- **이력:** 지난 확정 메뉴 목록.

(라우트 구조는 `app/` 아래에서 프로젝트 관례에 맞게.)

---

## 8. 하지 않는 것 (MVP 제외)

- 실제 주문·결제·배달앱 API 연동
- 개인 알레르기·칼로리 분석 등 부가 정보
- 복수 팀·조직 단위 관리자 대시보드 (필요 시 이후 스펙)

---

## 9. AI에게 개발 지시할 때 쓰는 체크리스트

아래를 만족하면 MVP 완료로 본다.

- [ ] 팀(또는 단일 팀) 단위로 **오늘 세션**이 존재하고, 팀원이 **메뉴 제안**할 수 있다.
- [ ] **1인 1표** 투표가 DB 수준(유니크 제약 등)에서 깨지지 않는다.
- [ ] **마감** 후 최다 득표(동점 정책 적용)로 **결정 메뉴**가 저장·표시된다.
- [ ] 모든 팀원이 **동일한 오늘의 메뉴**를 본다.
- [ ] **최근 확정 이력**을 목록으로 볼 수 있다.
- [ ] `AGENTS.md` 스택·보안 규칙을 지킨다.
